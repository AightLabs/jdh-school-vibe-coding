CHAPTER 01
바이브 코딩이란?什么是 Vibe 编程?
코딩을 몰라도, 아이디어만 있으면 시작할 수 있어요.不懂编程也没关系，只要有想法就能开始。
jdhschool.org · 바이브코딩 강의Vibe 编程课
개념 · CH1概念 · CH1

코딩 = 컴퓨터에게 주는 명령서编程 = 给电脑的命令书

컴퓨터는 빠르지만 스스로 알아서 하지 못해요. “이걸 이렇게, 그다음 저렇게”를 순서대로 정확히 적어줘야 하죠.电脑很快，但 不会自己判断。要 按顺序、精确地 写清「先这样，再那样」。

예전 방식过去的方式

문법을 하나하나 외워서 처음부터 직접 타이핑. 오타 하나에 멈춤.要背下所有语法，从头一个个手敲。一个错字就卡住。

바이브 코딩Vibe 编程

“이렇게 만들어줘”라고 말하고, 결과를 확인하며 고쳐 나감.用 话 说「帮我做成这样」，看结果再慢慢改。

핵심核心

“무엇을 만들고 싶은지”를 잘 설명하는 것이 실력이 돼요. (이 방식을 AI 연구자 안드레이 카파시가 ‘바이브코딩’이라 불렀어요.)把「想做什么」讲清楚，就是本事。(AI 研究者 Andrej Karpathy 把这种方式叫「Vibe Coding」。)

개념 · CH1概念 · CH1

그래도 원리는 알아야 해요但原理还是要懂

AI가 코드를 대신 써줘도, 어디를 어떻게 고쳐달라 말하려면 코드가 어떻게 동작하는지 알아야 합니다.就算 AI 帮你写代码，想说清 改哪里、怎么改，也得懂代码是怎么运作的。

  • 결과가 안 나올 때 무엇이 문제인지 짚을 수 있어요出问题时能指出 哪里错了
  • AI에게 더 정확하게 부탁할 수 있어요能 更准确地 给 AI 下指令
  • 작은 부분은 내 손으로 직접 고칠 수 있어요小地方能 自己动手 改
⚠️ 오해 금지别误会

“AI가 다 해주니 몰라도 된다”는 아니에요. AI는 똑똑한 조수, 방향을 정하고 판단하는 건 나예요. (개발자를 꿈꾼다면 특히!)不是「AI 全包了就不用懂」。AI 是 聪明的助手，定方向、做判断的是 我。(想当开发者更要懂!)

코딩의 4가지 기본 감각编程的 4 个基本直觉

개념 · CH2概念 · CH2

프로그래밍 언어는 어떻게 작동할까?编程语言是怎么运作的?

컴퓨터가 진짜 이해하는 건 0과 1(기계어)뿐. 사람이 읽기 쉬운 언어로 쓰면 번역기가 기계어로 바꿔줍니다.电脑真正懂的只有 0 和 1(机器语言)。人用易读的语言写，翻译器 再转成机器语言。

📚 컴파일 방식编译型

전체를 미리 통째로 번역. 빠름.
예: C, C++, Rust把整个程序 提前整体 翻译。快。
例: C, C++, Rust

🗣️ 인터프리터 방식解释型

한 줄씩 그때그때 번역. 결과를 바로 봄.
예: Python, JavaScript一行一行即时 翻译。马上看到结果。
例: Python, JavaScript

💡 비유打个比方

컴파일 = 책을 통째로 번역·출판 / 인터프리터 = 옆에서 실시간 통역. 우리가 쓸 JavaScript는 인터프리터라 브라우저가 바로 실행해요.编译 = 把整本书翻译出版 / 解释 = 在旁边实时口译。我们用的 JavaScript 是解释型，浏览器直接运行。

개념 · CH2概念 · CH2

Git이란 무엇인가?是什么?

Git은 코드의 “세이브 포인트”를 관리하는 도구. 저장(커밋)해두면 언제든 되돌릴 수 있어요.Git 是管理代码 「存档点」 的工具。存档(提交)后 随时能回退。

GitGitHub
무엇是什么세이브·버전관리 도구存档·版本管理 工具코드를 올려두는 인터넷 창고存放代码的 网络仓库
어디在哪내 컴퓨터 안我的电脑里인터넷(클라우드)互联网(云端)
💡 한 줄 정리一句话

Git으로 세이브(커밋) → GitHub로 보내기(푸시) → GitHub Pages로 공개(배포). GitHub를 쓰는 이유 = 무료 + 올리면 바로 웹사이트!用 Git 存档(commit) → 推送到 GitHub(push) → 用 GitHub Pages 公开(部署)。 用 GitHub 的理由 = 免费 + 一上传就是网站!

프롬프트 = AI에게 주는 설계도提示词 = 给 AI 的设计图

똑같은 AI라도 어떻게 부탁하느냐에 따라 결과가 완전히 달라져요. 좋은 프롬프트의 3원칙:同一个 AI，怎么问 决定结果天差地别。好提示词的 3 原则:

  • 구체적으로 — “예쁘게” ❌ → “배경 남색, 버튼 크고 둥글게, 글씨 흰색” ⭕要具体 — 「好看点」❌ → 「深蓝背景、按钮又大又圆、白字」⭕
  • 예시를 보여주기 — 원하는 화면·문구·목업을 함께 주기给例子 — 把想要的画面·文案·原型一起给它
  • 하나씩 시키기 — 한 번에 열 개 말고 작게 쪼개서一次一件 — 别一口气十件，拆小了做
비교 · CH4对比 · CH4

아쉬운 vs 좋은 프롬프트差的 vs 好的提示词

🙅 아쉬운差的

“게임 만들어줘”
→ AI가 뭘 만들지 몰라 엉뚱한 걸 만듦「做个游戏」
→ AI 不知道做什么，做出跑偏的东西

🙆 좋은好的

“가위·바위·보 버튼 3개, 누르면 컴퓨터가 랜덤으로 내서 승패를 크게 표시, 이기면 점수 +1”「石头剪刀布 3 个按钮，点了电脑随机出，大大显示胜负，赢了分数 +1」

💡 좋은 프롬프트의 구조好提示词的结构

역할(초보 학생이야) + 맥락(prd.md 참고) + 요청(무엇을) + 조건(순수 HTML/JS, 주석) + 출력형식(전체 코드로).角色(我是初学者) + 背景(参考 prd.md) + 要求(做什么) + 条件(纯 HTML/JS、加注释) + 输出形式(给出完整代码)。

개념 · CH5概念 · CH5

컨텍스트 = AI가 지금 아는 정보上下文 = AI 此刻知道的信息

AI는 내 머릿속을 못 봐요. 내가 준 정보(창)만큼만 봅니다. 이 창의 크기를 컨텍스트 창이라고 해요.AI 看不到我脑子里。它只看得到 我给的信息(窗口)。这个窗口的大小叫 上下文窗口。

  • 요즘 AI의 창은 대략 책 한 권 분량(예: 20만 토큰)까지 커졌어요如今 AI 的窗口大到 约一本书的量(例如 20 万 tokens)
  • 하지만 무한하지 않아요 — 대화가 길어지면 창이 꽉 차요但 不是无限的 — 对话变长，窗口会被塞满
  • 창이 차면 앞 내용을 잊거나 흐릿하게 기억해요塞满后会 忘掉或记不清 前面的内容
💡 그래서所以

큰 걸 만들수록 “무엇을 기억시킬지”를 내가 관리해야 해요. 이게 컨텍스트 엔지니어링이에요.做得越大，越要 由我来管理「让它记住什么」。这就是上下文工程。

방법 · CH5方法 · CH5

세 가지 습관 (도구 무관 원리)三个习惯(与工具无关的原理)

  • ① 문서화 — 중요한 결정·규칙을 파일로 적어두기 (context.md, prd.md)① 记录成文 — 把重要决定·规则 写进文件(context.md、prd.md)
  • ② 요약 — 세션(대화)이 길어지면 “지금까지 요약”을 만들어 파일에 저장② 做摘要 — 对话变长时，做一份 「到目前为止的摘要」 存进文件
  • ③ 재적재 — 새 세션을 시작할 때 그 요약을 먼저 불러오기(붙여넣기/참조)③ 重新载入 — 开新会话时先 把摘要读进来(粘贴/引用)
💡 왜 새 세션?为什么开新会话?

대화가 너무 길어지면 느려지고 흐려져요. 한 세션 = 한 목표로 짧게 유지하고, 기억은 파일에 남겨 이어가는 게 프로의 방식이에요.对话太长会变慢变糊。一个会话 = 一个目标，保持简短，记忆 留在文件里 接着做，这才是老手的方式。

개념 · CH6概念 · CH6

용어 빠르게 정리术语快速梳理

  • Wireframe(와이어프레임) — 색·꾸밈 없이 어디에 무엇이 있는지 뼈대만Wireframe(线框图) — 无颜色装饰，只画 什么在哪 的骨架
  • Mockup(목업) — 색·글꼴까지 입힌 실제로 보일 모습 (아직 안 눌림)Mockup(视觉稿) — 加上颜色·字体的 真实外观(还不能点)
  • Prototype(프로토타입) — 버튼이 눌러지고 화면이 넘어가는 시제품Prototype(原型) — 按钮 能点、画面能跳转 的样品
  • MVP — Minimum Viable Product, 핵심 하나가 진짜로 도는 최소 제품MVP — 最小可行产品，核心功能真的能跑 的最小产品
💡 왜 MVP부터?为何从 MVP 开始?

완벽을 노리면 못 끝내요. 핵심 하나가 도는 것을 먼저 만들고, 거기서부터 키웁니다.追求完美就做不完。先做出 一个核心能跑的东西，再往上加。
